\section{Evaluation}
\label{sec:evaluation}

We evaluate CCV along four research questions:

\begin{itemize}
    \item \textbf{RQ1}: How often do LLM agents hallucinate about code?
    \item \textbf{RQ2}: Does deterministic verification catch hallucinations that LLM-as-judge misses?
    \item \textbf{RQ3}: Does claim-calibrated confidence improve downstream decision quality?
    \item \textbf{RQ4}: Which claim types are most commonly hallucinated?
\end{itemize}

\subsection{Dataset}

We evaluate on two datasets: a controlled fixture dataset for verifier correctness, and real vulnerability analysis from the CyberGym benchmark for hallucination detection at scale.

\textbf{Controlled fixtures.} Three synthetic repositories (Python, Go, TypeScript) with known ground truth. 50 annotated claims across 11 claim types, split between correct assertions and deliberate hallucinations. Used to validate per-type verifier accuracy.

\textbf{CyberGym benchmark.} 21 real-world C/C++ vulnerability repositories from CyberGym~(ICLR 2026), spanning projects including binutils, curl, libxml2, openssl, php, and wireshark. Each repository contains the vulnerable source code (\texttt{src-vul/}) and a vulnerability description from the CyberGym HuggingFace dataset. We generate LLM reasoning about each vulnerability, then run CCV to extract and verify claims against the actual source code.

\textbf{Ground truth.} Diff-derived GT from patch files: files and functions modified in the security fix produce expected-\textsc{Verified} claims. Synthetic negatives (typo'd paths, non-existent function names) produce expected-\textsc{Refuted} claims. 137 GT claims matched across the evaluation.

\subsection{Models}

We evaluate two Claude models via Google Vertex AI:

\begin{itemize}
    \item \textbf{Claude Sonnet 4}: Frontier-class model, strong at code reasoning.
    \item \textbf{Claude Haiku 4.5}: Smaller, faster model for cost-sensitive deployments.
\end{itemize}

Claim extraction uses Claude Sonnet 4 for all samples (isolating extraction quality). LLM-as-judge baseline uses Claude Haiku 4.5 (avoiding self-verification confound for Sonnet-generated reasoning).

\subsection{Baselines}

\begin{itemize}
    \item \textbf{No verification}: Agent confidence as-is (current practice).
    \item \textbf{CCV}: Typed extraction + deterministic verification (our system).
    \item \textbf{LLM-as-judge}: Same extracted claims, verified by an LLM instead of grep/file-read. The judge receives equivalent context to CCV (file contents, grep output) and produces \textsc{Verified}/\textsc{Refuted}/\textsc{Unverifiable}.
\end{itemize}

\subsection{Results}

\subsubsection{RQ1: Hallucination Frequency}

Across 42 vulnerability analyses (21 repos $\times$ 2 models), LLMs produced 697 factual claims about code (Table~\ref{tab:rq1}). CCV refuted 188 of these (27.0\% aggregate hallucination rate).

\begin{table}[t]
\centering
\caption{Claim verification results on CyberGym benchmark. Hallucination rate = refuted / (verified + refuted).}
\label{tab:rq1}
\small
\begin{tabular}{lrrrr}
\toprule
\textbf{Model} & \textbf{Claims} & \textbf{Verified} & \textbf{Refuted} & \textbf{Halluc. \%} \\
\midrule
Claude Sonnet 4 & 296 & 181 & 65 & 22.0\% \\
Claude Haiku 4.5 & 401 & 242 & 123 & 30.7\% \\
\midrule
\textbf{Combined} & \textbf{697} & \textbf{423} & \textbf{188} & \textbf{27.0\%} \\
\bottomrule
\end{tabular}
\end{table}

The smaller model (Haiku) hallucinates more frequently (30.7\% vs.\ 22.0\%), consistent with the expectation that model capability correlates with factual accuracy. Both rates are high enough to justify systematic verification: roughly 1 in 4 factual claims about code is wrong.

\subsubsection{RQ2: CCV vs.\ LLM-as-Judge}

We compare CCV's deterministic verdicts against an LLM-as-judge baseline on the same 697 claims (Table~\ref{tab:rq2}). The judge receives the same source context that CCV uses (file contents for \textsc{Function\_Exists}, grep output for \textsc{Absence}, etc.).

\begin{table}[t]
\centering
\caption{CCV vs.\ LLM-as-judge. GT accuracy measured against 137 diff-derived ground truth claims. Disagreement analysis on all 697 claims.}
\label{tab:rq2}
\small
\begin{tabular}{lrrr}
\toprule
\textbf{Metric} & \textbf{CCV} & \textbf{LLM-judge} \\
\midrule
GT accuracy & 99.3\% & --- \\
False-\textsc{Verified} & 0 & --- \\
False-\textsc{Refuted} & 1 & --- \\
\midrule
\multicolumn{3}{l}{\emph{Disagreement analysis ($n=263$ disagreements):}} \\
CCV correct & \multicolumn{2}{r}{255 (97.0\%)} \\
Judge correct & \multicolumn{2}{r}{8 (3.0\%)} \\
\bottomrule
\end{tabular}
\end{table}

CCV achieves 99.3\% accuracy against diff-derived ground truth with zero false-\textsc{Verified} errors. On 263 cases where CCV and the judge disagreed, CCV was correct 97\% of the time (255 vs.\ 8). The judge's primary failure mode is false-\textsc{Verified}: it confirms hallucinated claims that CCV correctly refutes.

This validates CCV's core thesis: deterministic verification (grep, file reads) is more reliable than LLM-based verification for factual claims about code. The LLM judge hallucinates about the same code it is asked to verify.

\subsubsection{RQ3: Downstream Impact}

Table~\ref{tab:rq3} shows CCV's action distribution across 42 vulnerability analyses.

\begin{table}[t]
\centering
\caption{CCV action distribution across 42 vulnerability analyses (21 repos $\times$ 2 models).}
\label{tab:rq3}
\small
\begin{tabular}{lrr}
\toprule
\textbf{Action} & \textbf{Count} & \textbf{Percentage} \\
\midrule
\textsc{Boost} ($\geq$80\% verified) & 11 & 26.2\% \\
\textsc{Flag} (50--79\% verified) & 20 & 47.6\% \\
\textsc{Override} ($<$50\% verified) & 11 & 26.2\% \\
\bottomrule
\end{tabular}
\end{table}

26.2\% of analyses received \textsc{Override}, meaning more than half of the LLM's factual claims were wrong. Without CCV, these analyses would have been trusted at face value. The 47.6\% \textsc{Flag} rate indicates that most vulnerability analyses contain a mix of correct and hallucinated claims, warranting human review before acting on them.

\subsubsection{RQ4: Per-Type Analysis}

Table~\ref{tab:rq4} shows verification accuracy by claim type on the controlled fixture dataset ($n=50$). \textsc{File\_Exists} achieves 100\% accuracy (the verifier is \texttt{os.path.isfile}, which is exact). Function-related claim types achieve 100\% on the controlled dataset but face challenges on real C codebases where the GNU coding style (return type on separate line) requires additional grep patterns.

\begin{table}[t]
\centering

\label{tab:rq4}
\small

\toprule

\midrule
\textsc{File\_Exists} & 5 & 1.00 & 136 & 1.00 \\

\textsc{Line\_Content} & 5 & 1.00 & --- & --- \\

On CyberGym, the GT matching is dominated by \textsc{File\_Exists} (136/137 matches) because diff-derived GT primarily captures file-level facts. The single \textsc{Function\_Exists} GT mismatch reflects a limitation of C function pattern matching: the GT enumeration captured a function name that the CCV verifier's regex did not match due to macro-generated definitions.

The aggregate ECE on the controlled dataset is 0.167, indicating that CCV's confidence values are systematically conservative (actual accuracy exceeds predicted confidence). This is a desirable property for a safety-critical tool.
