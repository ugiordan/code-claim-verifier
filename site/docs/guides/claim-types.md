# Claim Types

CCV ships with 14 built-in claim types organized into four categories. Each type has a dedicated verifier that uses deterministic tools to check the claim against the actual codebase.

## File Claims

### FILE_EXISTS

Checks whether a file exists at a given path in the repository.

| Field | Value |
|-------|-------|
| **Parameters** | `path` (string, required): relative file path |
| **Verification method** | `os.path.isfile` |
| **Confidence** | 0.99 |
| **Triggers on reasoning like** | "The file `src/config.py` contains..." or "In `utils/auth.go`..." |

The verifier resolves the path relative to the repo root and checks for path traversal before testing existence.

### LINE_CONTENT

Checks whether a specific line in a file contains expected content.

| Field | Value |
|-------|-------|
| **Parameters** | `path` (string, required), `line` (integer, optional), `content` (string, required) |
| **Verification method** | File read + substring match |
| **Confidence** | 0.95 |
| **Triggers on reasoning like** | "Line 42 of `model.py` calls `torch.load()`" |

Reads the file, checks the specified line number, and does a substring match. If the line number is out of range, the claim is REFUTED.

### FILE_CLASSIFICATION

Checks whether a file matches a classification: test, vendored, or production.

| Field | Value |
|-------|-------|
| **Parameters** | `path` (string, required), `classification` (string, required) |
| **Verification method** | Path regex matching |
| **Confidence** | 0.80 |
| **Triggers on reasoning like** | "This is a test file" or "The code is in a vendored directory" |

Uses path patterns to classify: `test[s_/]`, `_test.`, `spec[s_/]` for tests; `vendor/`, `third_party/`, `node_modules/` for vendored. Everything else is production.

### GENERATED_OR_VENDORED

Checks whether a file is auto-generated or vendored (not hand-written source).

| Field | Value |
|-------|-------|
| **Parameters** | `path` (string, required), `expected` (boolean, default true) |
| **Verification method** | File header inspection + path check |
| **Confidence** | 0.85 |
| **Triggers on reasoning like** | "This is an auto-generated file" or "The code is generated by protoc" |

Checks for vendored paths (`/vendor/`, `/third_party/`, `/node_modules/`) and reads the first 1000 bytes looking for markers like `DO NOT EDIT`, `AUTO-GENERATED`, `@generated`, etc.

## Function Claims

### FUNCTION_EXISTS

Checks whether a function or method is defined in the codebase.

| Field | Value |
|-------|-------|
| **Parameters** | `name` (string, required), `file` (string, optional) |
| **Verification method** | Grep for language-specific function definition patterns |
| **Confidence** | 0.85 |
| **Triggers on reasoning like** | "The `parseConfig` function in `config.go`..." |

Uses language-aware regex patterns: `def name(` for Python, `func name(` for Go, `function name` for JS/TS, etc. If `file` is provided, searches only that file. Otherwise searches the whole repo.

### FUNCTION_CALLED

Checks whether a function is called (invoked) somewhere in the codebase.

| Field | Value |
|-------|-------|
| **Parameters** | `name` (string, required), `expected` (boolean, default true) |
| **Verification method** | Grep for call sites, excluding definition sites |
| **Confidence** | 0.65 |
| **Triggers on reasoning like** | "torch.load() is called at..." or "This function is never invoked" |

Greps for `name\s*\(` across the repo, then subtracts lines that match the function definition pattern. What remains are call sites. The `expected` flag controls polarity: set to `true` if the LLM claims the function IS called, `false` if it claims it's NOT called.

### HAS_CALLERS

Checks whether a function has any callers (is referenced or invoked).

| Field | Value |
|-------|-------|
| **Parameters** | `name` (string, required), `expected` (boolean, default true) |
| **Verification method** | Grep for call sites, excluding definition sites |
| **Confidence** | 0.65 |
| **Triggers on reasoning like** | "This function has no callers" or "The endpoint is reachable from..." |

Mechanically identical to FUNCTION_CALLED but semantically distinct. FUNCTION_CALLED is about a specific invocation, HAS_CALLERS is about reachability.

## Dependency Claims

### IMPORT_EXISTS

Checks whether a module or package is imported in the codebase.

| Field | Value |
|-------|-------|
| **Parameters** | `module` (string, required), `file` (string, optional) |
| **Verification method** | Grep for language-specific import patterns |
| **Confidence** | 0.80 |
| **Triggers on reasoning like** | "The file imports `os.path`" or "There is no import of `subprocess`" |

Uses language-aware patterns: `import module` / `from module import` for Python, `"module"` for Go, `import ... from 'module'` / `require('module')` for JS/TS, etc.

### PACKAGE_VERSION

Checks a dependency's version against lockfiles and manifests.

| Field | Value |
|-------|-------|
| **Parameters** | `package` (string, required), `version` (string, optional) |
| **Verification method** | Lockfile parsing |
| **Confidence** | 0.90 |
| **Triggers on reasoning like** | "The project uses PyYAML version 5.3.1" |

Searches for the package in, in order: `go.sum`, `go.mod`, `requirements.txt`, `poetry.lock`, `package-lock.json`. Parses each format natively. If the package is found, checks whether the actual version matches the claimed version.

### DEPENDENCY_TYPE

Checks whether a dependency is direct or transitive.

| Field | Value |
|-------|-------|
| **Parameters** | `package` (string, required), `dep_type` (string, required) |
| **Verification method** | Manifest file check |
| **Confidence** | 0.85 |
| **Triggers on reasoning like** | "PyYAML is a transitive dependency" or "requests is a direct dependency" |

Checks manifest files (`go.mod`, `requirements.txt`, `package.json`, `Pipfile`, `pyproject.toml`) for the package name. If found in a manifest, it's classified as direct. If not found in any manifest, it's assumed transitive.

### CVE_AFFECTS_VERSION

Checks whether a CVE affects the installed version of a package.

| Field | Value |
|-------|-------|
| **Parameters** | `cve` (string, optional), `package` (string, required), `version` (string, optional) |
| **Verification method** | Currently returns UNVERIFIABLE |
| **Confidence** | 0.0 |
| **Triggers on reasoning like** | "CVE-2024-1234 affects PyYAML < 6.0" |

!!! note
    This claim type always returns UNVERIFIABLE because CVE version range checking requires an external advisory database. It exists as a typed slot so the extraction prompt captures CVE-related claims rather than dropping them.

## Code Claims

### ABSENCE

Checks that something does NOT exist in the codebase.

| Field | Value |
|-------|-------|
| **Parameters** | `pattern` (string, required), `scope` (string, default "repo") |
| **Verification method** | Fixed-string grep (inverted logic) |
| **Confidence** | 0.60 |
| **Triggers on reasoning like** | "There is no input validation" or "The code never checks for null" |

Runs `grep -F` for the pattern. If zero matches are found, the claim is VERIFIED (the absence is confirmed). If matches are found, it's REFUTED. The `scope` parameter can be `file`, `directory`, or `repo`.

The lower confidence (0.60) reflects that absence claims are inherently harder to verify. Grep can confirm a literal string is absent, but the LLM might be describing a concept rather than a literal pattern.

### MITIGATION_EXISTS

Checks whether a security mitigation or safeguard exists at a specific location.

| Field | Value |
|-------|-------|
| **Parameters** | `mitigation` (string, required), `file` (string, required), `pattern` (string, optional) |
| **Verification method** | File read + content check |
| **Confidence** | 0.70 |
| **Triggers on reasoning like** | "Input sanitization is applied at line 58 of handler.go" |

Reads the specified file and line, checking that the line contains non-trivial content (more than 5 characters). This is a rough heuristic: it confirms something exists at the claimed location, but can't semantically verify that it's actually a mitigation.

### ENTRY_POINT

Checks whether a function or endpoint is an entry point (externally reachable).

| Field | Value |
|-------|-------|
| **Parameters** | `name` (string, required), `type` (string, optional) |
| **Verification method** | Grep for entry point registration patterns |
| **Confidence** | 0.65 |
| **Triggers on reasoning like** | "This HTTP endpoint is externally reachable" or "The CLI command exposes..." |

Searches for framework-specific patterns based on the `type` parameter:

- **http/rest/api**: `HandleFunc`, `.GET(`, `@app.route`, `@router.`, `http.Handle`, etc.
- **grpc/rpc**: `RegisterServer`, `pb.*Server`
- **cli/command**: `cobra.Command`, `argparse.`, `click.command`, `flag.`

If no type is specified, searches for all pattern categories.

## Confidence levels summary

| Confidence | Claim Types |
|-----------|-------------|
| 0.99 | FILE_EXISTS |
| 0.95 | LINE_CONTENT |
| 0.90 | PACKAGE_VERSION |
| 0.85 | FUNCTION_EXISTS, DEPENDENCY_TYPE, GENERATED_OR_VENDORED |
| 0.80 | FILE_CLASSIFICATION, IMPORT_EXISTS |
| 0.70 | MITIGATION_EXISTS |
| 0.65 | FUNCTION_CALLED, HAS_CALLERS, ENTRY_POINT |
| 0.60 | ABSENCE |
| 0.00 | CVE_AFFECTS_VERSION (always UNVERIFIABLE) |

These confidence values are used as weights during calibration. Higher confidence means the verifier's result has more influence on the final score.
